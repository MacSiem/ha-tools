Audit of ha-storage-monitor.js CSS and visual consistency:

---

### 1. CARD VS CONTAINER

- Uses `.card` wrapper extensively for main container and sub-containers:
  - Main panel wrapper: `<div class="card">` (line ~345)
  - Various `.card`, `.card-container`, `.reports-card`, `.export-card` classes styled in CSS (lines ~360-370)
- No usage of `.container` class anywhere in the template or CSS.
- **Count**: `.card` used multiple times, `.container` not used.
- **Verdict**: Correct usage per Bento design system.

---

### 2. BENTO TOKENS

- CSS almost exclusively uses `--bento-*` CSS variables for colors, borders, backgrounds, shadows.
- Hardcoded colors found only in inline styles or very few places:
  - In `_renderOverview` treemap cells: inline style `background:${c.color}` where `c.color` is hex like `#9c27b0`, `#ff9800`, etc. (line ~500)
  - In `_renderAddonsAndIntegrations` table rows, e.g.:
    - Status color: `style="color:${a.state === 'started' ? '#4caf50' : '#9e9e9e'}"` (line ~620)
    - Size bar background: `background:#4caf50` (line ~628)
    - Integration source badges: background and color hardcoded rgba and hex (line ~650)
    - Backup size bar: `background:#9c27b0` (line ~700)
    - File size bar: background `#ff9800` or `#3b82f6` (line ~770)
    - Cleanup suggestions border-left and background: `var(--bento-error)`, `var(--bento-error-light)`, etc. (line ~800)
- These hardcoded colors correspond to Bento palette colors (green, purple, orange, blue) and are consistent with Bento tokens.
- No raw RGB or HSL colors found in CSS blocks.
- **Verdict**: Mostly uses Bento tokens; hardcoded colors are limited, purposeful, and consistent with Bento palette.

### 3. DARK MODE

- Has explicit `@media (prefers-color-scheme: dark)` block (line ~480-490) redefining Bento tokens for dark mode.
- Uses `color-scheme: light dark;` on `:host` (line ~470).
- Bento tokens are redefined for dark mode, e.g. `--bento-bg`, `--bento-card`, `--bento-text`, `--bento-border`, shadows.
- No dark mode missing or incomplete tokens detected.
- **Verdict**: Proper dark mode support implemented.

### 4. RESPONSIVE

- Has `@media (max-width: 768px)` and `@media (max-width: 480px)` blocks (line ~520-560).
- Adjusts padding, grid columns, flex directions, font sizes, and gauge sizes for smaller screens.
- No `360px` breakpoint found.
- Tabs have horizontal scroll overflow with `overflow-x: auto` and `-webkit-overflow-scrolling: touch` for mobile (line ~530).
- `.disk-gauge` flex direction switches to column on very small screens (line ~550).
- No obvious content that would overflow horizontally on mobile due to scrollable tabs and flexible grids.
- **Verdict**: Responsive design present and adequate for common breakpoints (768px, 480px). No 360px breakpoint but not critical.

- Count of inline style attributes in template literals: approx. 20+ occurrences.
- Top 5 longest inline style strings with approximate line numbers:

### 8. ACCESSIBILITY

- Color contrast:
  - Uses Bento tokens which are designed for good contrast.
  - Status colors (green #4caf50, red #f44336, orange #ff9800) have good contrast on white backgrounds.
  - Text colors use `--bento-text` and `--bento-text-secondary` which adapt in dark mode.
- Aria labels:
  - Interactive elements like tabs and buttons lack explicit `aria-label` or roles.
  - Tabs are buttons with `data-tab` but no `aria-selected` or `role="tab"` attributes.
  - Refresh button has text "🔄 Refresh" but no aria-label.
  - Tables have headers but no ARIA roles or summaries.
- Keyboard focus:
  - Buttons and tabs are native `<button>` elements, so keyboard focus is supported.
- Missing:
  - No aria attributes on tabs for accessibility (e.g., `role="tablist"`, `role="tab"`, `aria-selected`).
  - No aria-live regions for loading or error messages.
- **Verdict**: Color contrast is good; accessibility semantics and ARIA attributes are missing or minimal.
