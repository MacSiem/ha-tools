Audit of ha-smart-reports.js CSS and visual consistency:

### 1. CARD VS CONTAINER

- The main wrapper uses `.card` class consistently (line ~210 in the template literal inside `_render()` method).
- No usage of `.container` class found anywhere in the CSS or template.
- `.card` is used per Bento design system conventions.

**Count:**
- `.card` wrappers: 1 (main card div)
- `.container` wrappers: 0

**Verdict:** Correct usage of `.card` wrapper.

### 2. BENTO TOKENS

- The CSS heavily uses `--bento-*` CSS variables for colors, borders, backgrounds, shadows, radii, transitions, and fonts.
- Examples: `var(--bento-primary)`, `var(--bento-bg)`, `var(--bento-border)`, `var(--bento-text-secondary)`, etc.
- Hardcoded colors found (not using Bento tokens):

  - In `.bar-fill` colors array (line ~350): `#4caf50`, `#66bb6a`, `#81c784`, `#a5d6a7`, `#c8e6c9`, `#e8f5e9`, `#fff9c4`, `#ffcc80`, `#ffab91`, `#ef9a9a`
  - Domain colors in `_renderSystem()` (line ~670): `#4caf50`, `#8bc34a`, `#ffc107`, `#2196f3`, `#ff9800`, `#00bcd4`, `#9c27b0`, `#795548`, `#607d8b`, `#e91e63`, `#ff5722`
  - Button active box-shadow color: `rgba(59, 130, 246, 0.25)` (line ~160)
  - Tab active background: `rgba(59, 130, 246, 0.04)` (line ~140)
  - Inline styles for icons and emojis use no colors but emojis themselves.
  - Inline styles in empty state icon: `opacity:0.5` (line ~370)
  - Inline styles for bar-fill min-width: `30px` (line ~290)

- These hardcoded colors mostly correspond to semantic colors (green, orange, blue, red) and are consistent with Bento palette but not using tokens.

**Verdict:** Mostly Bento tokens used, but some hardcoded semantic colors for bars and domain colors exist.

### 4. RESPONSIVE

- Multiple `@media` breakpoints present:

  - `@media (max-width: 768px)` (lines ~350-380 and ~410-430)
  - `@media (max-width: 480px)` (lines ~390-400 and ~440-460)
  - `@media (max-width: 360px)` (lines ~470-480)

- Responsive adjustments include:

  - Padding and font size reductions
  - Grid columns reduced (e.g., `.stats-grid` from multiple columns to 2 or 1)
  - Tabs become scrollable horizontally with smaller gaps and font sizes
  - Chart container height reduced on small screens
  - Flex-wrap adjustments for tabs

- No obvious content that would overflow horizontally on mobile due to:

  - Tabs have `overflow-x: auto` and `-webkit-overflow-scrolling: touch`
  - Grids collapse to 1 column on narrow screens
  - Tables reduce font size and padding on small screens

**Verdict:** Responsive design is well implemented with appropriate breakpoints and no visible overflow issues.

### 8. ACCESSIBILITY

- Color contrast:

  - Uses Bento tokens which are designed for good contrast.

  - Active tab text is white on `var(--bento-primary)` blue background, which is good contrast.

  - Disabled buttons have opacity 0.4 and cursor not-allowed, visually distinct.

  - Inline colors for trends use semantic colors (green/red/orange) but also have text labels, which helps.

- Missing aria labels:

  - Tab buttons have no `aria-selected` or `role="tab"` attributes.

  - The tabs container has no `role="tablist"`.

  - Buttons like export CSV/JSON have no aria-labels; text is visible so acceptable but aria-labels would improve screen reader clarity.

  - The period select dropdown has no label element or aria-label; it uses a visible label "Period" in the UI? No, it uses a select with id `periodSelect` but no label element associated.

- Keyboard accessibility:

  - No explicit keyboard handling for tabs (e.g., arrow keys).

- Overall, basic accessibility is present by visible text, but ARIA roles and labels are missing for tabs and some controls.

**Verdict:** Minor accessibility issues due to missing ARIA roles and labels on tabs and controls.
