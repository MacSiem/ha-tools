Audit of ha-frigate-privacy.js CSS and visual consistency:

---

**1. CARD VS CONTAINER**

- The main wrapper uses `<div class="card">` (line ~650 in _buildHTML).
- No `.container` class usage found anywhere in the CSS or HTML templates.
- **Result:** Uses `.card` wrapper correctly per Bento design system.

  - `.btn-extend` border: `1px solid rgba(59,130,246,0.3)` and background `rgba(59,130,246,0.08)` and hover background `rgba(59,130,246,0.15)`
  - `.btn-extend` color: `var(--bento-primary)` (good)

- **Result:** Mostly uses Bento tokens, but some hardcoded rgba colors for status and hover highlights.

**3. DARK MODE**

- The editor component defines dark mode Bento variables explicitly.
- **Result:** Dark mode support is present via Bento tokens and explicit dark mode CSS in editor.

**7. CONSISTENCY**

- Buttons use `.btn` base class with modifiers `.btn-primary`, `.btn-success`, `.btn-danger`, `.btn-secondary`, `.btn-quick`, `.btn-full`.
- Cards use `.card` and `.camera-card` with consistent border-radius, shadows, and hover states.
- Tabs use `.tab-btn` with active and hover states.
- Tables and lists (history, schedule) use consistent spacing, font sizes, and colors.
- Status badges use `.status-badge` with color-coded classes `.status-running`, `.status-stopped`, `.status-unknown`.
- Toast messages use `.toast` with modifiers `.toast-success`, `.toast-error`, `.toast-info`.
- Overall styling is consistent in colors, spacing, font sizes, and interactive states.
- **Result:** UI elements are styled consistently throughout.

---

**8. ACCESSIBILITY**

- Color contrast:

  - Uses semantic colors for statuses (green, red, gray) with semi-transparent backgrounds and solid text colors.
  - Text colors use Bento tokens which are designed for good contrast.
  - Privacy banner uses red text on light red background, likely sufficient contrast.
  - Buttons have hover and focus states with color changes.
  - Disabled buttons have opacity 0.5 and cursor not-allowed.

- Aria labels:

  - Tabs have no explicit `aria-selected` or roles, but buttons have clear text.
  - Schedule toggle buttons have `title` and `aria-label` attributes (line ~800 in _buildScheduleTab).
  - Other interactive elements (camera cards, quick buttons) lack explicit aria labels.
  - Some buttons use emoji icons without aria-labels (e.g., camera status icons).
  - No keyboard focus styles explicitly defined, but buttons are native elements and should be focusable.
  - No role attributes on tablist or tab panels.

- Potential improvements:

  - Add `role="tablist"` on tabs container and `role="tab"` on tab buttons with `aria-selected`.
  - Add aria-labels on camera cards and quick pause buttons for screen readers.
  - Ensure focus styles are visible (not overridden).

- **Result:** Basic accessibility present, but missing some ARIA roles and labels for full screen reader support.
